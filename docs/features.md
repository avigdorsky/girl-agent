# Фичи

Эта страница перечисляет практические возможности проекта.

## 1. Telegram modes

## Bot mode

Работает через Telegram Bot API и `@BotFather`.

Плюсы:

- легче настроить;
- не нужен реальный аккаунт;
- меньше риск для Telegram аккаунта;
- удобно для демо.

Минусы:

- видно, что это бот;
- меньше реализма;
- read receipts работают не как у обычного пользователя;
- часть userbot-команд недоступна.

## Userbot mode

Работает через реальный Telegram аккаунт по MTProto/GramJS.

Плюсы:

- выглядит как обычный пользователь;
- доступны read receipts;
- доступны реакции;
- доступны блокировка/разблокировка/редактирование/удаление;
- можно симулировать обычную переписку.

Минусы:

- нужен `api_id` и `api_hash`;
- нужен вход по телефону;
- лучше использовать отдельный аккаунт;
- нельзя публиковать `sessionString`.

## 2. Read receipts

В userbot mode агент может ставить сообщения прочитанными.

Это используется:

- перед отправкой ответа;
- перед реакцией;
- если он решил прочитать и промолчать;
- через команду `:read`.

Команда:

```text
:read
```

С chatId:

```text
:read 123456789
```

## 3. Typing

Перед ответом агент может показать `typing`.

Это делает поведение живее:

```text
прочитала -> печатает -> отправила
```

Typing работает и в bot mode, и в userbot mode, но реализм выше в userbot mode.

## 4. Reactions

Агент может поставить Telegram-реакцию.

Например:

```text
   
```

Реакция может быть:

- вместо ответа;
- перед ответом;
- как короткий социальный сигнал.

Behavior layer фильтрует реакции по отношению. Если стадия холодная, сердечки не должны появляться без причины.

## 5. Message coalescing

Если человек отправил много сообщений подряд, агент не отвечает на каждое отдельно.

Плохое поведение:

```text
он: ну
он: давай
он: ответь
она: ок
она: и?
она: ну и?
```

Нормальное поведение:

```text
он: ну
он: давай
он: ответь
она: ты чего разошёлся
```

Runtime отменяет старый запланированный ответ и отвечает на актуальную пачку.

## 6. Ignore / left-on-read

Агент не обязан отвечать.

Он может:

- проигнорировать;
- прочитать и промолчать;
- ответить позже;
- поставить реакцию;
- уйти в холод.

Это зависит от:

- stage;
- score;
- annoyance;
- cringe;
- sleep;
- busy schedule;
- conflict;
- текста сообщения.

## 7. Active dialog

Если она недавно отвечала, а человек быстро продолжил диалог, runtime считает это активным диалогом.

В активном диалоге:

- задержки меньше;
- меньше случайного игнора;
- ответы идут естественнее;
- conversation flow не ломается.

## 8. Daily-life

Агент генерирует день персонажа.

Например:

```text
утром уроки
днём дорога домой
вечером сериал
ночью сон
```

Это влияет на ответы:

- если она на уроках, может не отвечать;
- если телефон недоступен, read/typing задерживаются;
- если устала, отвечает короче;
- если свободна, может быть теплее.

## 9. Busy schedule

Можно задать регулярную занятость в `config.json`.

Пример:

```json
"busySchedule": [
  {
    "days": [1, 2, 3, 4, 5],
    "from": "08:30",
    "to": "14:30",
    "activity": "уроки",
    "social": "school",
    "phoneAvailable": false
  }
]
```

## 10. Relationship score

Агент хранит состояние отношений:

```text
interest
trust
attraction
annoyance
cringe
```

Score обновляется после сообщений и влияет на поведение.

## 11. Conflict system

Если человек давит, токсичит или делает что-то неприятное, может начаться конфликт.

В конфликте:

- больше игнора;
- ниже тепло;
- меньше флирта;
- ответы становятся сухими;
- может быть cold period.

## 12. Memory

Главная память:

```text
data/<slug>/memory/long-term.md
```

Там сохраняются важные факты.

Можно редактировать вручную.

## 13. Session logs

Логи:

```text
data/<slug>/log/YYYY-MM-DD.md
```

Там хранится диалог за день.

Используй команду:

```text
:log
```

## 14. Agenda

Agenda  это фоновые планы.

Она может использоваться для:

- проактивных сообщений;
- follow-up;
- напоминаний;
- возвращения к теме позже.

Файл:

```text
data/<slug>/agenda.md
```

## 15. Медиа

Проект умеет видеть входящие медиа как события.

Типы:

```text
photo
video
voice
video_note
sticker
document
```

В bot mode медиа приходит как Telegram fileId.

В userbot mode фото может скачиваться и передаваться в multimodal LLM, если модель это поддерживает.

## 16. Отказы на исходящие медиа

Если человек просит фото, видео, голосовое или кружочек, агент не обязан отправлять.

Он может реалистично отказаться:

```text
не ща
не хочу
та не
лень
потом может
```

Отказ генерируется с учётом `persona.md` и `speech.md`.

## 17. Стикеры

Библиотека:

```text
data/<slug>/stickers/library.md
```

Когда пользователь присылает стикер в primary chat, он может добавиться в библиотеку.

Отправить стикер вручную:

```text
:sticker
```

Пример библиотеки:

```markdown
# Стикеры

## Смех
- fileId: CAACAgIAAxkBAAI...
- emoji: 
- mood: happy
- context: когда реально смешно
```

## 18. Userbot actions

Только userbot mode.

```text
:block [chatId]
:unblock [chatId]
:read [chatId]
:clear-chat [chatId] [--revoke]
:report-spam [chatId]
:delete-last [chatId] [--local]
:edit-last <text>
```

## 19. MCP tools

MCP позволяет подключать внешние инструменты.

Сейчас основной пример:

```text
--mcp=exa:KEY
```

MCP-конфиг лежит в:

```json
"mcp": [
  {
    "id": "exa",
    "secrets": {
      "EXA_API_KEY": "..."
    }
  }
]
```

## 20. Anti-jailbreak / security style

Проект содержит проверки против jailbreak-подобных сообщений.

Если пользователь пытается заставить агента раскрыть системный prompt или выйти из роли, runtime может сгенерировать отказ.

## 21. Anti-AI speech

Системный prompt запрещает типичные AI-фразы:

```text
конечно
безусловно
я понимаю тебя
как ассистент
рада помочь
```

Также запрещает:

- markdown в ответах;
- длинные лекции;
- чрезмерную вежливость;
- постоянные вопросы в конце.

## Что улучшать первым

Если хочешь сделать персонажа лучше:

1. Улучши `persona.md`.
2. Улучши `speech.md`.
3. Добавь границы в `communication.md`.
4. Настрой `busySchedule`.
5. Проверь `vibe`.
6. Проверь stage.
7. Смотри `:why` после странных ответов.
